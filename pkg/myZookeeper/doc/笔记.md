# 理论知识

## CAP

一个分布式系统必然会存在一个问题：**因为分区容忍性（partition tolerance）的存在，就必定要求我们需要在系统可用性（availability）和数据一致性（consistency）中做出权衡** 。这就是著名的 `CAP` 定理。

- 分区容错性
  - 分区是分布式系统中的通信中断，即两个节点之间断开连接或连接临时延迟。 分区容错性是指无论系统中节点之间的通信出现多少故障，集群都必须持续工作。
- 可用性
  - 发出数据请求的任何客户端都会得到响应，即使一个或多个节点宕机。 另一种关于可用性的说法是：分布式系统中的所有工作节点都可返回任何请求的有效响应，而不会发生异常。
- 一致性
  - 所有客户端可以同时看到相同的数据，无论它们连接到哪个节点。 要做到这一点，每当数据写入一个节点时，就必须立即将其转发或复制到系统中的所有其他节点，然后写操作才被视为"成功"。

## BASE

即使无法做到强一致性，但分布式系统可以根据自己的业务特点，采用适当的方式来使系统达到最终的一致性。BASE理论由：`Basically Avaliable` 基本可用、`Soft state` 软状态、`Eventually consistent` 最终一致性组成。

- **基本可用(Basically Available)**：基本可用是指分布式系统在出现故障的时候，允许损失部分可用性，即保证核心可用。例如，电商大促时，为了应对访问量激增，部分用户可能会被引导到降级页面，服务层在该页面只提供降级服务。
- **软状态(Soft State)**： 软状态是指允许系统存在中间状态，而该中间状态不会影响系统整体可用性。分布式存储中一般一份数据至少会有多个副本，允许不同节点间副本同步的延时就是软状态的体现。
- **最终一致性(Eventual Consistency)**： 最终一致性是指系统中的所有数据副本经过一定时间后，最终能够达到一致的状态。弱一致性和强一致性相反，最终一致性是弱一致性的一种特殊情况。

一句话概括就是：平时系统要求是基本可用，运行有可容忍的延迟状态，但是，无论如何经过一段时间的延迟后系统最终必须达成数据是一致的。

## ACID

数据库管理系统（DBMS）在写入或更新资料的过程中，为保证事务（transaction）是正确可靠的，所必须具备的四个特性：原子性（Atomicity，或称不可分割性）、一致性（Consistency）、隔离性（Isolation，又称独立性）、持久性（Durability）

-   **原子性（Atomicity）：**原子性实际描述的是同一个客户端对于多个操作之间的限制，这里的原子表示的是不可分割，原子性的效果是，假设有操作集合{A,B,C,D,E}，执行后的结果应该和单个客户端执行一个操作的效果相同。
    -   一个事务（transaction）中的所有操作，要么全部完成，要么全部不完成，不会结束在中间某个环节。事务在执行过程中发生错误，会被恢复（Rollback）到事务开始前的状态，就像这个事务从来没有执行过一样。
-   **一致性（Consistency）：**表示写入的资料必须完全符合所有的预设规则，这包含资料的精确度、串联性以及后续数据库可以自发性地完成预定的工作。这个名词在不同语境中含义会不同，但可能又有联系，这就可能让我们陷入混乱。
    -   数据复制时，副本间具有一致性，这个一致性应该指不同副本状态的一致；一致性Hash，这是一种分区算法，个人理解是为了能够在各种情况下这个Hash算法都可以以一致的方式发挥作用；CAP定理中的一致性指的是后面要介绍的一个特殊的内部一致性，称为“线性一致性”。
-   **隔离性（Isolation）：**隔离性实际上解决的问题是多个事务作用于同一个或者同一批数据时的并发问题，防止由于交叉执行而导致数据的不一致。
    -   一个完美的事务隔离，在每个事务看来，整个系统只有自己在工作，对于整个系统而言这些并发的事务一个接一个的执行，也仿佛只有一个事务，这样的隔离成为“可序列化（Serializability）”。当然，这样的隔离级别会带来巨大的开销，因此出现了各种各样的隔离级别，进而满足不同场景的需要。
-   **持久性（Durability）:** 只要事务完成，不管发生任何问题，都不应该发生数据丢失。即，事务处理结束后，对数据的修改就是永久的，即便系统故障也不会丢失。
    -   从理论上讲，如果是单机数据库，起码数据已被写入非易失性存储（至少已落WAL），分布式系统中数据被复制到了各个副本上，并受到副本Ack。但实际情况下，也未必就一定能保证100%的持久性。

## ZAB

ZAB（ZooKeeper Atomic Broadcast）协议是ZooKeeper中用于数据复制和一致性保证的协议。它是ZooKeeper的核心组件，用于实现分布式数据的可靠复制和节点之间的协调。

ZAB协议的主要目标是提供高可用性、一致性和顺序性的数据复制。下面是ZAB协议的关键特点和工作原理：

-   数据复制：ZAB协议通过将写操作转化为事务，并将事务进行广播，实现数据在集群中的复制。Leader节点接收客户端的写请求，并将写操作转化为一个事务（包含多个写操作），然后将事务广播给其他节点。
-   原子广播：广播是ZAB协议的核心机制，用于将事务传播给其他节点。ZAB协议保证了事务在整个集群中的原子性广播，即事务要么被全体节点接受并执行，要么被全体节点拒绝。
-   数据一致性：ZAB协议保证了所有节点在相同的顺序下接收和执行事务，从而实现了数据的一致性。每个节点按照事务的广播顺序逐个执行事务，以保持数据的一致性。
-   Leader选举：ZAB协议还包括Leader选举机制，用于在集群中选择一个Leader节点。Leader节点负责接收客户端的写请求，并协调其他节点的数据复制。Leader选举确保了集群中的唯一Leader，并提供了高可用性和故障容忍性。
-   快照机制：ZAB协议还使用快照机制来加速节点的启动和恢复过程。快照是对整个ZooKeeper数据树的状态的完整拷贝，可以用于在节点启动时进行快速恢复。

## 工作原理

ZooKeeper是一个分布式协调服务，用于管理和协调分布式系统中的各种任务和资源。它基于ZAB（ZooKeeper Atomic Broadcast）协议实现了高可用、一致性和顺序性的数据复制和协调。

下面是ZooKeeper的工作原理的概述：

1.  集群架构：ZooKeeper由多个服务器节点组成，形成一个ZooKeeper集群。集群中的每个节点都是对等的，没有主从之分。节点之间通过ZAB协议进行通信和数据复制。
2.  数据模型：ZooKeeper提供一个类似文件系统的数据模型，称为ZooKeeper树（ZooKeeper Tree），也称为ZooKeeper命名空间（ZooKeeper Namespace）。树中的每个节点称为znode，它可以包含数据和子节点。
3.  数据复制：ZooKeeper使用ZAB协议进行数据的复制和一致性保证。ZAB协议将数据复制分为两个阶段：Leader选举和原子广播。
    -   Leader选举：集群中的节点通过选举机制选择一个Leader节点，Leader节点负责处理客户端的读写请求和协调其他节点。
    -   原子广播：Leader节点接收客户端的写请求，并将写操作转化为事务进行原子广播给其他节点。其他节点按照广播的顺序执行这些事务，从而保持数据的一致性和顺序性。
4.  客户端访问：客户端可以连接到ZooKeeper集群，并与其进行交互。客户端可以执行各种操作，如创建、读取、更新和删除znode，以及注册监听器以接收znode的变更通知。
5.  事件通知：ZooKeeper可以监测并通知客户端关于znode的变更。当某个znode发生变化时（例如数据修改、子节点变更等），ZooKeeper将通知相关的监听器，以便客户端可以做出相应的响应。
6.  数据一致性：ZooKeeper提供强一致性的数据模型。即使在面对网络分区或节点故障的情况下，ZooKeeper也能保持数据的一致性，并提供可用性和持久性的保证。

ZooKeeper的工作原理主要依赖于Leader选举、ZAB协议和基于树形结构的数据模型。它通过提供高可用性、一致性和顺序性的分布式协调服务，为分布式系统提供了可靠的基础设施。

## 选举

ZooKeeper中Leader的选择是通过Leader选举机制完成的。Leader节点负责处理客户端的读写请求并协调其他节点的数据复制。下面是ZooKeeper选择Leader的详细过程：

1.  节点状态：每个ZooKeeper节点都有一个状态，可以是以下三种之一：LOOKING、FOLLOWING和LEADING。
    -   LOOKING：节点处于Leader选举状态，它会发送选举消息并等待选举结果。
    -   FOLLOWING：节点跟随当前的Leader节点，接收并复制Leader的事务日志。
    -   LEADING：节点自己成为Leader，并负责处理客户端请求和协调其他节点。
2.  选举轮次：当一个节点启动时或者当前的Leader节点失效时，Leader选举过程开始。每个节点会创建一个临时顺序节点（EPHEMERAL_SEQUENTIAL）作为候选人，并开始发送选举消息。
3.  选举协议：ZooKeeper使用ZAB协议的Leader选举协议来进行选举。选举协议包括以下步骤：
    -   节点进入LOOKING状态，并生成一个唯一的选举ID。
    -   节点将选举ID和自己的信息（如IP地址和端口号）作为选举消息广播给其他节点。
    -   接收到选举消息的节点比较选举ID，选择具有最大选举ID的节点作为新的Leader候选人。
    -   如果有多个节点具有相同的最大选举ID，则比较节点的ZXID（ZooKeeper Transaction ID），即事务ID。具有最大ZXID的节点成为新的Leader候选人。
    -   其他节点收到新Leader候选人的选举消息后，更新自己的状态为FOLLOWING，并将新Leader候选人的信息保存下来。
4.  选举结果：当选举过程结束时，只有一个节点会成为新的Leader。其他节点将成为新Leader的FOLLOWING节点。
5.  同步数据：新的Leader开始处理客户端请求，并将数据变更通过ZAB协议进行广播给FOLLOWING节点，使得整个集群达到一致状态。

总结起来，ZooKeeper的Leader选举过程是通过节点之间的选举消息和比较选举ID的方式完成的。具有最大选举ID和最大ZXID的节点成为新的Leader。选举过程保证了集群中只有一个Leader，从而确保系统的高可用性和一致性。
